# 久留米飲み屋ナビ

福岡県久留米市・西鉄久留米駅周辺(一番街・二番街・文化街周辺エリア)のバー・居酒屋・コンカフェ・シーシャ・アミューズメントポーカーバー・スナック・キャバクラなど、飲み屋を幅広く紹介する静的サイト。エースハイ合同会社の収益プロジェクト。

- 会社側のプロジェクト概要: `projects/kurume-bar-navi/README.md`(このリポジトリではなく会社ワークスペース側)
- 公開サイト(GitHub Pages): デプロイ後に `https://<GitHubユーザー>.github.io/kurume-bar-navi/` で公開予定
- 費用: **ゼロ円構成**。独自ドメインなし(GitHub Pagesの無料サブドメインを使用)、有料API・有料プランは一切使用していません。

## これは何か

外部APIに依存せず、**Node.js標準モジュールのみ**で `data/*.json` から静的HTMLを生成するシンプルな構成です(フレームワーク・データベース・サーバーなし)。GitHub Actionsが `main` ブランチへのpushをトリガーに自動でビルド・GitHub Pagesへのデプロイを行います。

```
data/*.json (店舗・エリア・業態データ)
      │  node scripts/build.js
      ▼
dist/ (静的HTML一式、sitemap.xml、robots.txt)
      │  GitHub Actions (.github/workflows/deploy.yml)
      ▼
GitHub Pages で公開
```

## なぜこの構成か(2026-07-17時点の判断)

- 当初はNext.js + Vercel + Supabase + Stripeによる動的構成(店舗によるセルフサーブの編集申請・PR枠決済)を検討したが、①費用ゼロ方針への転換、②まずは情報収集・掲載のMVPを優先、という社長判断を受けて、claim申請・決済フローは次フェーズに切り出した。
- 動的機能が不要になったため、`fukuoka-poker-navi` と同様に**静的サイト+GitHub Pages**へ回帰。ただし店舗ページが多数(数十〜将来的に数百)になる想定のため、fukuoka-poker-navi(単一HTML+クライアントサイド描画)とは異なり、**店舗ごとに実体のあるHTMLファイルをビルド時に生成**する方式にした(SEO上、店舗名で検索された際にクロール・インデックスされやすくするため。JSON-LD構造化データも各ページに個別出力)。
- Google Places API等の有料/規約制限のあるAPIには依存しない(詳細は「データソースとその収集方法」を参照)。

## データソースとその収集方法

`data/venues.json` の店舗情報は、**常設の自動巡回スクレイパーではなく、開発時にWeb検索・公開ページの確認を通じて人力寄りに収集した一次リスト**です(2026-07-17時点)。主な情報源:

- Yahoo!検索(検索結果内のYahoo!マップ情報。店名・業態・最寄り駅からの徒歩分数・大まかな営業時間帯を参照)
- 食べログ・Retty・ホットペッパーグルメ・ぐるなび等のグルメサイトの一覧ページ(店名の参照のみ。口コミ・写真・評点は転載していない)
- 各店舗の公式サイト・Instagram・X等のSNS(公開情報)
- **福岡県料飲業生活衛生組合連合会「久留米文化街社交組合(筑後ブロック)組合員一覧」**(https://f-ryoin.jp/union/kurume-bunkagai ) — 業界団体が公表している組合員(店舗)の名称・住所・電話番号。スナック・クラブ系の情報源として特に信頼性が高い
- 求人サイト(バイトル・体入ショコラ等)に掲載されている店舗の事実情報(店名・所在エリア・業態)の要約

**転載していないもの**: 各サイトの口コミ文章、評点(星の数)、写真。店舗ごとに「情報源」リンクを掲載し、最新情報は出典元または店舗公式情報を確認するよう案内している。

**Google Places API不使用の理由**: Google Maps Platform利用規約を確認した結果、(a) 「掲載・ディレクトリサービスでの利用」を明示的に禁止する条項(3.2.3(d))、(b) 店名・住所・営業時間・評点・写真等の恒久キャッシュ/自前DB保存が実質不可という条項(Places API専用条項14.3等)に抵触するため、主データソースとして不採用とした(会社ワークスペース側 `projects/kurume-bar-navi/README.md` 参照)。

**継続的な自動収集は現時点では未実装**。店舗データを追加・更新する場合は、上記と同じ方法(Web検索・公式情報の確認)で `data/venues.json` を人力で編集し、コミット・PRを作成する運用とする。GitHub Actionsが自動化しているのは「データが更新されたらビルド・公開する」部分のみで、「データを取得する」部分は自動巡回ボットにしていない(利用規約違反リスクを避けるため)。

## タグによる絞り込み機能

社長方針により、「タグによる細かい絞り込み検索」をサイトの核となる機能と位置付けている(2026-07-17)。

- 各店舗の `tags` に、設備・遊べる要素(ダーツ・ビリヤード・カラオケ・生演奏/ライブ等)や、飲み物の専門性(ウイスキー専門/充実等)、料理ジャンルの細分化(もつ鍋・餃子・海鮮/魚介・焼肉・焼き鳥等)、席・利用シーン(個室あり・カウンター席あり・一人客歓迎・大人数対応等)を付与している。
- **タグは必ず出典(店舗個別ページ)に基づいて付与し、推測では付けない**。ホットペッパーグルメ・Retty等の店舗個別ページの free text 説明や「お店の特長」表から、明示的な言及がある場合のみタグ化している。「個室 なし」のように否定形で項目名だけが出現するケースを誤って肯定と読み取らないよう、キーワード直後に「なし/不可」が続く場合は除外する処理を入れている(`scripts/` 配下には含めていない一時的な抽出スクリプトで作業し、結果を `data/venues.json` に反映する運用。抽出ロジック自体は将来再利用する場合に備えて開発部内で保管)。
- カテゴリページ(`/categories/{id}/`)・エリアページ(`/areas/{id}/`)には、そのページに実在する店舗のタグだけを選択肢にしたチェックボックス式の絞り込みUIを表示する(複数選択でAND絞り込み、外部ライブラリ不使用のvanilla JS)。
- `/tags/` にタグ一覧、`/tags/{タグ名}/` に各タグの該当店舗一覧を静的生成している(JS無効環境やクローラーでも閲覧・インデックス可能)。
- 店舗個別ページのタグは、そのタグの一覧ページへのリンクになっている。

## デザイン方針・ビジュアル(2026-07-21更新)

社長要望「文字ばかりで店が何か分からない・圧が強くて次に進む気が失せる」の解消を主眼に、以下を実装した。

- **モバイルファースト**: `assets/style.css` は基本スタイルをスマホ向けに書き、`min-width: 600px / 900px` のメディアクエリでタブレット・PC向けに拡張する構成。タップ領域は44px前後を確保。
- **ビジュアルヒーロー(店舗ページ最上部)**: 各店舗ページの一番上に、業態別の大きなSVGアイコン+業態カラーのグラデーション背景のバナー(`.venue-hero`)を配置し、その中に店名(大)・業態・エリア・最寄駅からの距離を載せている。**開いた瞬間に「何の店か・どこか」が分かる**状態を全店で実現している。
- **セクション分け+余白**: 店舗ページの情報を、アイコン付き見出し(`.section-heading`、📋店舗情報 / 🗺地図・アクセス / 📷写真 / 🔗情報源 / ✉️関係者の方へ)でセクション分けし、セクション間の余白を増やして「文字の壁」に見えないようにした。
- **一覧ページのカードグリッド化**: カテゴリ/エリア/タグ/検索結果の各一覧を、縦一列リストから**カードグリッド(タイル状。モバイル1〜2列、タブレット3列、PC4列)**に変更。各カード(`.venue-card`)は上部に業態カラーの小さなヘッダー(アイコン+業態名)、下に店名・エリア・代表タグ3個までに情報を絞り、スキャンしやすくした。
- **写真は「店の公式ソース」限定で掲載(2026-07-21、社長判断で方針一部変更)**: **第三者グルメサイト(食べログ・ホットペッパー・Retty・ぐるなび等)の写真は引き続き一切使わない**(明確な著作権侵害+各サイトのスクレイピング規約違反のため)。掲載してよいのは**その店自身が公式に発信している写真(公式サイト・公式Instagram)のみ**。加えて、実写真の代わりの視覚的アイデンティティとして、業態別自作アイコン(`CATEGORY_ICONS`/`CATEGORY_COLORS`。外部画像・フォント不使用)と業態カラーのビジュアルヒーローを全店で使っている。
  - **公式サイト画像はホットリンク(自サイトに保存しない)**: 公式サイトがあり `og:image` 等がある店舗は、その画像を**店のサーバー上のURLを直接参照する `<img>`(ホットリンク)**で表示している。**画像ファイルを自サイトにダウンロード・保存(rehost)することは一切していない**(複製・保存を発生させず侵害の度合いを最小化するため)。実装は `scripts/build.js` の `OFFICIAL_PHOTOS`/`officialPhotoHtml`。対象は4店(bar-remember, bar-oshu-kitchen-alma, izakaya-sumibi-sakagura-kita, izakaya-kiseki-tebasaki)。各画像URLは、当サイトのドメインを Referer に付けたクロスオリジン要求で HTTP 200 + `image/*` を返すことを実測で確認済み(=リファラによるホットリンクブロックなし)。ただし実ブラウザでの最終描画は未検証(この環境では確認できない)。
    - **除外例(2026-07-21 レビュー部判断)**: bar-lampsquare は画像が `cdn.r-corona.jp`(Recruit系CDN)上にあり、公式サイト自体が Recruit の店舗ページ作成サービス owst.jp(RestaurantBOARD)製。Recruit は今回禁止対象のホットペッパーグルメの親会社であり、「クリーンな公式ソース限定」の線引きを濁すため対象外とした(写真なし=ビジュアルヒーローのまま)。公式サイトを新規に採用する際は、そのサイト/画像CDNが禁止対象の第三者グルメサイト系列でないかを確認すること。
  - **【著作権リスク階層の線引き(重要)】**: 現状の写真掲載は、**(1) 店の公式ソース限定、(2) 画像を自サイトに保存せずホットリンク、(3) 出典明記、(4) 削除依頼導線あり**、という**最低リスク形態**に限定している。この形態である限り、個別の弁護士確認は不要という前提で運用している。一方で、**次のいずれかに踏み込む場合は別のリスク階層になり、実施前に必ず専門家(弁護士)確認を経ること**: (a) 写真を自サイトにダウンロード・保存(rehost)する、(b) 公式ソースであっても写真掲載を大規模(多数店舗)に展開する、(c) 第三者グルメサイトの写真に手を出す(これは現状明確に禁止)、(d) **有料掲載枠(PR枠)・広告枠を導入する場合、非同意店舗のロゴ・写真表示との併存可否を再検討する**(2026-07-22 レビュー部指摘。掲載許諾を得ていない店舗のロゴと有料枠が同一画面に混在すると、「当サイトと掲載店に提携・推奨関係がある」という誤認リスクが質的に上がるため、導入を検討する時点でレビュー部の再判定を必ず経ること)。これらは複製・データベース化の度合い、あるいは営業上の誤認の度合いが上がり、権利者からの請求リスクが質的に変わるため。
  - **公式Instagram埋め込み**: 店舗公式アカウント本人の投稿を Instagram公式の埋め込み機能で表示(下記参照)。
  - **すべての写真に出典表示+削除依頼案内**: 公式サイト画像・Instagram埋め込みのいずれも、提供元(公式サイト/公式アカウント)へのリンクと、「掲載を希望されない店舗様は[連絡先]までご連絡ください」という削除依頼の案内文を写真の近くに表示している(連絡先は現状 `CONTACT_EMAIL` のプレースホルダー。**削除依頼が届く可能性が上がるため、本設定=実受信できるメールアドレスへの差し替えが急務。経営管理オフィス経由で社長に対応を諮る事項**)。
  - **店舗ロゴの表示(2026-07-22、社長要望により実装)**: 公開144店を全数調査し、**店舗自身(またはチェーン運営元)の公式サイトにロゴ画像がある30店**について、一覧カードと店舗ページのビジュアルヒーローの**業態アイコンをロゴ画像に差し替えて**表示している。写真と同じく**ホットリンクのみ**で、**ロゴ画像を自サイトに保存(rehost)することは一切していない**。詳細は下記「店舗ロゴ」節を参照。ロゴが無い店舗は従来どおり業態別アイコン/ビジュアルヒーローが視覚的アイデンティティを担う。
  - **「写真を見る」外部リンクボタン**: 各店舗ページに、出典サイト(公式Instagram・Retty・ホットペッパー等、写真が充実している傾向のあるサイトを優先)への外部リンクボタンを設置。実際の写真は出典元で見てもらう導線。
  - **Instagram公式埋め込み(該当店舗のみ)・公式プロフィールリンク**: 下記参照。
  - **Googleマップの地図埋め込み+外部リンク**: 下記参照。

## 店舗ロゴ(2026-07-22)

社長要望の最優先項目として、店舗ごとのロゴ表示を実装した。実装は `scripts/build.js` の `VENUE_LOGOS` / `venueIconSlotHtml()` / `venueLogoCreditHtml()`、スタイルは `assets/style.css` の「店舗ロゴ」節。

- **対象**: 公開144店を全数調査し、**店舗自身(またはチェーン運営元)の公式サイトにロゴ画像が確認できた30店**。第三者グルメサイト(食べログ・ホットペッパー・Retty・ぐるなび)およびそのページ作成サービス(owst.jp / gorp.jp / r-corona.jp)由来の画像は**採用しない**(採用時に機械チェック済み)。
- **表示場所**: (1) 一覧ページの店舗カード(カテゴリ/エリア/タグ/検索/トップの掲載店舗)、(2) 店舗ページのビジュアルヒーロー。いずれも**業態アイコン(自作SVG)の枠をそのままロゴ画像に差し替える**方式で、ロゴが無い店舗は従来の業態アイコンのまま。
- **ホットリンク限定**: 公式サイト画像(`OFFICIAL_PHOTOS`)と同じ方針で、**画像ファイルは自サイトに保存しない**。`<img src="公式サイト上のURL" loading="lazy">` で直接参照する。
- **見え方の担保**: 正方形のアバター枠 + `object-fit: contain` + 背景色で、縦横比の違うロゴでも崩れないようにしている。ロゴは透過PNGが多いため既定の背景は**白**。ただし**白抜き(白一色)のロゴは白背景だと不可視**になるため、実際に画像を取得して白背景・濃色背景それぞれに合成し目視確認したうえで、該当する3店(`izakaya-kuimonoya-wan` / `izakaya-toriichizu` / `izakaya-shanghai-shuka`)は `bg: "dark"` を指定して濃色背景にしている。ロゴを追加する際も同様に**両方の背景での見え方を確認**すること。
- **フォールバック**: 画像の読み込みに失敗した場合は `onerror` で業態アイコンを表示に戻す(枠の装飾も外す)。店舗ページでは、ロゴが出せなかったのに出典表記だけが残らないよう出典行も隠す。
- **出典・削除依頼**: 店舗ページのヒーロー直下に「ロゴ画像: <提供元(公式サイト)へのリンク>のものを直接参照して表示しています(当サイトには保存していません)」+ 削除依頼の連絡先(`CONTACT_EMAIL`)を表示する。一覧カードは煩雑になるため出典表記を出さない(店舗ページ側で担保)。
- **実測(curl、2026-07-22)**: 30件すべて、当サイトのGitHub Pagesドメインを Referer に付けたクロスオリジン要求で **HTTP 200 + `Content-Type: image/*`** を返すことを確認済み(=リファラによるホットリンクブロックなし)。ヘッドレスChromeでの描画確認も実施し、カード・ヒーローでのロゴ表示、白抜きロゴの視認性、`onerror` フォールバックの動作を確認した。ただし**実機(スマホ実ブラウザ)での最終確認は未実施**。
- **サイズへの配慮**: 表示サイズに対し極端に重い画像は採用しない。`izakaya-shiroichi` は原寸が約930KBあるため、Wix標準のリサイズ済みURL(約55KB)を参照している。
- **リスク階層**: 上記は写真掲載と同じ「(1)公式ソース限定・(2)自サイトに保存しない・(3)出典明記・(4)削除依頼導線あり」という最低リスク形態の範囲内。**ロゴ画像をrehostする、第三者グルメサイトのロゴを使う等は別リスク階層**であり、実施前に必ず専門家確認を経ること(上記「著作権リスク階層の線引き」参照。有料掲載枠を導入する場合の再検討事項(d)も同項に記載)。
- **商標帰属・提携否認の表示(2026-07-22 レビュー部指摘により必須化)**: ロゴ掲載の本質的リスクは商標権侵害そのものよりも、**営利ディレクトリがロゴを並べることで「掲載店と提携・公認関係がある」と誤認させること**(不正競争防止法2条1項1号・2号の領域)にある。今回のロゴには三代目鳥メロ(ワタミ)・磯丸水産(SFP)・くいもの屋わん(オーイズミフーズ)・とりいちず・戦国焼鳥 家康・かこみ庵など**全国チェーンの著名ブランド**が複数含まれるため、**全ページのフッター(`DISCLAIMER`)と `/about/` の両方**に「店舗名・ロゴ・商標は各権利者に帰属する」「掲載店舗との間に提携・協賛・推奨・公認等の関係はない」旨を明記している。**ロゴ表示を継続する限り、この2か所の記載を削除しないこと。**

### ロゴ・写真の採否基準(ホスト元の線引き)

`bar-lampsquare` を除外し(PR #9)、`izakaya-kuimonoya-wan`(画像が運営会社オーイズミフーズのドメイン上)を採用した(PR #11)判断の線引きは以下のとおり。

- **判定基準は「そのホストが禁止対象の第三者グルメサイト系列に属するか」であって、「店の独自ドメインかどうか」ではない。**
- **禁止**: ホットペッパーグルメ・食べログ・Retty・ぐるなび、およびそれらのページ作成サービス/CDN(`owst.jp`、`gorp.jp`、`r-corona.jp`、`cdn.r-corona.jp` 等)。公式サイトを名乗っていても、実体がこれらのサービス製であれば対象外とする。
- **可**: 店舗の独自ドメイン、**チェーン運営会社自身のサイト**(例: オーイズミフーズ `oizumifoods.co.jp`、ワタミ `torimero.com`)、**中立的なCMS・サイトビルダー/画像ホスト上の配信**(例: `momo.cmosite.com`、`static.wixstatic.com`)。中立ホストの場合は、**その店の公式サイト自身がそのホストからアセットを配信していること**を実際にページを取得して確認する(かこみ庵で確認済み)。
- 判定に迷う場合は採用しない。除外した候補とその理由は、収集時の調査メモに残す運用とする。

## Googleマップ: 地図iframe埋め込み(全店展開)+外部リンク(2026-07-21)

**経緯**: (1) 当初、キーレス地図iframe(`/maps/embed?pb=<自作base64>`)を実装したが、pb を住所の base64 で自作したのは本物のGoogle形式(座標/場所ID)ではなく無効で、品質管理部の実測で全件 HTTP 404 + X-Frame-Options: SAMEORIGIN となり撤回(2026-07-19)。当時「最終エンドポイントは XFO を返さず frameable と確認済み」と書いたが、これは別URLでの確認を一般化した誤記載だった(再発防止: 以後「確認済み」と書く前に本番と同一URLを実測する)。(2) その後、全店で「Googleマップで開く」外部リンク(`/maps/search/?api=1&query=...`、実測 HTTP 200)に一本化。(3) 社長判断で、`maps.google.com/maps?q=<住所>&output=embed` 形式のiframeを代表3店舗でテスト → 社長が実機(スマホ/PC)で地図表示を確認 → **住所が番地まで明確な全店舗に展開(2026-07-21)**。

**現在の実装**:
- **住所が番地まで明確な店舗(公開144件中134件。2026-07-22の住所精緻化で121件から増加)**: `maps.google.com/maps?q=<URLエンコード住所>&output=embed` の地図iframeを店舗ページに埋め込み(`loading="lazy"`・レスポンシブ・`border:0`・`allowfullscreen`)。
- **住所が曖昧な店舗(10件)**: iframeは出さず外部リンクのみ。
- **全公開店舗**: iframe の有無にかかわらず、その直下に「Googleマップで開く」外部リンク(`/maps/search/?api=1&query=...`)を必ず設置(iframeが表示されない環境のフォールバック)。
- 実装は `scripts/build.js` の `mapOutputEmbedUrl`/`mapSearchLink`/`mapSectionHtml`。

**実測事実(curl、2026-07-20)**: `output=embed` URLは初段 HTTP 301 + X-Frame-Options: SAMEORIGIN(→`www.google.com/maps/embed?origin=mfe&pb=...` へリダイレクト)、最終先 HTTP 200・XFOなし。ブラウザは通常リダイレクトの XFO を無視し最終応答のみ評価するため実ブラウザでは frameable になり得る。**実ブラウザでの最終描画はテスト3店舗で社長が実機確認済み**(同形式の横展開は同じ挙動)。開発環境(コマンドライン)では実描画は検証できない。

**規約面**: `output=embed`(キーレス埋め込み)・外部リンクは、いずれもAPIキー必須の Maps Embed API(Platform ToS適用、directory禁止条項3.2.3(d)(iii)を含む)とは別で、消費者向け Google Maps 追加利用規約 + Geoガイドラインの適用対象。Geoガイドラインは「ウェブサイトにGoogleマップを埋め込むだけなら許可不要」「テキスト/ボタンでのリンクも歓迎」と明記し、directory禁止条項は消費者向け規約には存在しない。**なお、この規約解釈の最終妥当性はレビュー部が「規約/公式形式/専門家確認のいずれかを踏まえて再判定する」との申し送りがあり、レビュー部確認待ち。**

## 条件の組み合わせ検索(2026-07-22)

**「Googleマップに原理的にできない、条件の組み合わせ検索」をこのサイトの主機能と位置づける**(社長方針)。トップページも業態一覧中心から「探す」中心の構成に変更した。

- **「いま営業中の店だけ」**: 端末の現在時刻・曜日で営業中の店に絞り込む(曜日・時刻を任意指定することも可能)。深夜営業(翌朝まで)は前日の枠として正しく判定する。
- **予算(夜)**: 〜2000円 / 2000〜3000円 / 3000〜4000円 / 4000円〜。予算が幅で書かれている店(例「2001〜3000円」)は、幅が重なるバケットすべてに該当させる。
- **支払い**: カード可 / キャッシュレス可 / 現金のみ。
- **喫煙**: 禁煙 / 分煙 / 喫煙可。
- **チャージ**: お通し・チャージなし(飲み屋で最も知りたい情報の一つで、かつグルメサイト以外では調べにくい差別化要素)。
- 上記は**既存のエリア×業態×タグの絞り込みと自由に組み合わせられる**(軸をまたぐ場合はAND)。エリア別・業態別・タグ別ページのウィジェットにも同じ条件が入る。
- トップページの「予算から探す」「こだわり条件から探す」チップは `/search/?budget=2000-3000` のようなURLクエリ付きリンクで、検索ページ側が読み取って初期条件として適用する(`?open=now` / `?area=` / `?category=` / `?budget=` / `?pay=` / `?smoke=` / `?charge=` / `?tags=`。カンマ区切りで複数指定可)。

### 設計: 機械可読データはビルド時に生成する

`data/venues.json` は**出典の表記をそのまま残した日本語の文字列を正**とし(人が読んで検証できる形を1か所に保つため)、絞り込みに必要な機械可読データは `scripts/build.js` のパーサ(`parseSchedule` / `parseClosedDays` / `parseBudget` / `paymentTokens` / `smokingToken` / `isChargeFree`)がビルド時に生成して、店舗カードの `data-open` / `data-budget` / `data-pay` / `data-smoke` / `data-charge` 属性に出力する。絞り込み自体は外部ライブラリ不使用のvanilla JS(`FILTER_SCRIPT`)。

- `data-open` は「曜日(日=0),開始分,終了分」の3つ組をセミコロン区切りにしたもの。深夜は24:00超(例: 翌3:00=1620分)で表現する。
- **パースできなかった店舗は、その条件での絞り込み対象から外れるだけ**で、店舗ページの表示(文字列)は従来どおり出る。ビルドログにパース成功件数と失敗した文字列を出力するので、データ追加時はここを確認すること(2026-07-22時点: 営業時間130件中129件をパース、失敗1件は `bar-jinga45`「20:00頃〜(要確認)」)。
- 「20:00〜LAST」のように**終了時刻が公開されていない**枠は、深夜5:00までを暫定の枠として扱い、店舗ページのバッジに「営業中(終了時刻は要確認)」と明示する(閉店時刻を勝手に断定しない)。JSON-LDの `openingHoursSpecification` にもこの枠は出力しない。
- 「第2・第4木曜」のような隔週の定休日は、週次の休みとしては扱わない(誤って毎週休みにしないため)。
- 絞り込みウィジェットには「その項目の情報を確認できた店舗のみが対象」と明記している。

### 店舗ページの表示

- **チャージ・お通しを専用のハイライトブロック**で店名直下に表示(`chargeCalloutHtml`)。「お通しなし」と明記されている店はそれ自体が価値のある情報なので、緑のバッジ付きで強調する。ただし**時間帯・条件により料金が発生する店(例: 「Cafe Timeは無料、Bar Timeは550円」)を「なし」と断定しない**よう、チャージ・お通し・席料の金額表記が併記されている場合は「なし」扱いにしない(`isChargeFree`)。
- 店舗情報テーブルに 定休日・予算(夜/昼)・チャージ・席数・支払い方法・喫煙・予約 の行を追加(値がある店舗のみ表示)。
- ヒーロー部に**営業中/営業時間外バッジ**をクライアントサイドで表示(端末の時刻で判定)。
- JSON-LDに `openingHoursSpecification` を追加(144店中124店)。`priceRange` も予算から補完する。

## エリア・業態・タグを組み合わせた絞り込み検索(2026-07-17)

- `/search/` に、全公開店舗を対象に**エリア・業態・タグを同時に組み合わせて絞り込める**統合検索ページを新設した。
- 既存の `/areas/{id}/`・`/categories/{id}/`・`/tags/{タグ名}/` はSEO(エゴサーチ・検索流入)のため個別ページとして維持しつつ、それぞれのページ内にも「他の軸」の絞り込みチェックボックスを追加した(例: `/categories/bar/` にエリアの絞り込み、`/areas/ichibangai/` に業態の絞り込みを追加。すでに固定されている軸(そのページ自体の軸)は表示しない)。
- 絞り込みは `scripts/build.js` の `filterWidgetHtml`/`FILTER_SCRIPT` で実装。エリア・業態は選択したいずれかに一致(OR)、タグは選択した全てを含む(AND)、軸をまたぐ場合はAND(エリア AND 業態 AND タグ)。外部ライブラリ不使用のvanilla JSで、対象の店舗一覧に実在する値だけを選択肢にする。
- JSが無効な環境でも、各個別ページ(`/areas/`・`/categories/`・`/tags/`)単体の閲覧・全件表示は引き続き可能(絞り込みチェックボックスが機能しないだけで、ページ自体は静的に生成された全件が表示される)。

## タグによる絞り込み機能

社長方針により、「タグによる細かい絞り込み検索」をサイトの核となる機能と位置付けている(2026-07-17)。

- 各店舗の `tags` に、設備・遊べる要素(ダーツ・ビリヤード・カラオケ・生演奏/ライブ等)や、飲み物の専門性(ウイスキー専門/充実等)、料理ジャンルの細分化(もつ鍋・餃子・海鮮/魚介・焼肉・焼き鳥等)、席・利用シーン(個室あり・カウンター席あり・一人客歓迎・大人数対応等)を付与している。
- **タグは必ず出典(店舗個別ページ)に基づいて付与し、推測では付けない**。ホットペッパーグルメ・Retty等の店舗個別ページの free text 説明や「お店の特長」表から、明示的な言及がある場合のみタグ化している。個室・座敷・カウンター・飲み放題・大人数対応(宴会)・コース料理の7タグは、フリーテキストのゆるいキーワード一致による誤検出(例:「個室：なし」を「個室あり」と誤認識する等)が過去に発生したため、「ラベル+値」の構造化欄(ホットペッパーグルメの「お店の特長」表)を正規表現で厳密に照合する方式に切り替えている(構造化欄が見つからない場合はタグを付与しない)。抽出ロジック自体は`scripts/`配下には含めていない一時的な抽出スクリプトで作業し、結果を `data/venues.json` に反映する運用(将来再利用する場合に備えて開発部内で保管)。
- `/tags/` にタグ一覧、`/tags/{タグ名}/` に各タグの該当店舗一覧を静的生成している(JS無効環境やクローラーでも閲覧・インデックス可能)。
- 店舗個別ページのタグは、そのタグの一覧ページへのリンクになっている。

## 店舗データの追加・更新方法

1. `data/venues.json` に店舗オブジェクトを追加する。スキーマ:
   ```json
   {
     "id": "category-短い英数字スラッグ",
     "name": "店舗名",
     "category": "bar|izakaya|concafe|shisha|poker|snack|kyabakura のいずれか",
     "area": "ichibangai|nibangai|bunkagai|eki-shuhen のいずれか",
     "address": "住所(不明な場合は市区町村レベルで可)",
     "walk": "西鉄久留米駅徒歩◯分 等、不明ならnull",
     "hours": "営業時間(出典の表記のまま。例: 「月〜木 17:00〜翌0:00 / 金土 17:00〜翌2:00」)、不明ならnull",
     "closedDays": "定休日(例: 「水曜」「不定休」)、不明ならnull",
     "phone": "電話番号、不明ならnull(050の転送番号は入れない)",
     "priceRange": "価格帯、不明ならnull",
     "budgetDinner": "夜の予算(例: 「3001〜4000円」)、不明ならnull",
     "budgetLunch": "昼の予算、不明ならnull",
     "seats": "席数・席構成、不明ならnull",
     "payment": "支払い方法(例: 「カード可、電子マネー不可、QR可」)、不明ならnull",
     "smoking": "喫煙可否(例: 「全席禁煙(店外に喫煙所)」)、不明ならnull",
     "charge": "チャージ・お通し・席料(例: 「お通し代330円(税込)」「お通し代なし」)、不明ならnull",
     "reservation": "予約可否・貸切条件、不明ならnull",
     "tags": ["任意のタグ"],
     "sources": [{ "label": "情報源の名称", "url": "情報源URL" }]
   }
   ```
   - `hours` / `closedDays` / `budgetDinner` / `payment` / `smoking` / `charge` は**絞り込み検索に使われる**。値は出典の日本語表記のまま入れてよく、機械可読化は `scripts/build.js` のパーサが行う(上記「条件の組み合わせ検索」参照)。ビルドログにパース結果が出るので、追加後は必ず確認すること。
   - **ファイルは「1店舗=1行」のフォーマットを維持する**(差分を読みやすく保つため)。
   - タグは表記ゆれを作らないこと(一人客での利用は `一人客歓迎`、個室は `個室あり` に統一済み)。
2. `node scripts/build.js` でローカルビルドし、`npm run serve` で `http://localhost:8080/` を確認(ただし内部リンクは `/kurume-bar-navi/` 始まりのためローカルではリンク遷移が正しく動かない点に注意。ページ単体の表示確認や `dist/` の中身確認には問題ない)。
3. ブランチを切ってコミット・PR作成。品質管理部のチェック→レビュー部のGO判定を経て `main` にマージされると、GitHub Actionsが自動でビルド・GitHub Pagesへ反映する。

## 掲載可否・フェーズ分類のガバナンス基準(2026-07-23、レビュー部の再発防止提言)

掲載/非掲載、フェーズ1(公開)/フェーズ2(非公開)の判断を属人的にしないための基準。**新規店舗を追加するとき・既存店を見直すときに必ず適用する。**

### フェーズ1/2の境界は「カテゴリ名」ではなく「実態(接待性)」で判定する

- **接待型課金・接待接客がある → フェーズ2(非公開)**。具体的には、キャストが客席に付いて接客する、またはキャストドリンク・指名料・同伴・シングルチャージ等の**接待を前提とした課金**がある店。表向きのカテゴリが居酒屋・コンカフェであっても非公開にする(例: 2026-07-23に `izakaya-nyanko-sakaba` 娘娘酒場=実態ガールズバー、`concafe-platinum-seven`、`concafe-axia` を移動)。
- **カウンター越しの接客かつ通常の飲食料金体系のみ → フェーズ1可**。ただし**業態表記は実態どおり**にする(コンカフェをバーと偽らない等)。
- この基準は、カテゴリallowlist(`PUBLISHED_CATEGORIES`)だけでは拾えない「カテゴリ内の個別店」を対象にする。実装は `scripts/build.js` の `PHASE2_VENUE_IDS`(店舗単位で公開ビルドから完全除外。dist・sitemap・検索・タグ・一覧・JSON-LDのどこにも出さない=スナック・キャバクラ23店と同じ漏洩ゼロ水準)。データ自体は将来の再判断のため `data/venues.json` に残す。追加/削除時のタイポは build 時の整合性チェック(存在しないIDなら `[warn]`)で検出する。

### 新規店舗追加時の必須チェック

`data/venues.json` に店舗を足す前に、次を確認する:

1. **接待性の判定**(上記)。少しでも接待型課金・接待接客の疑いがあれば、フェーズ1に入れず `PHASE2_VENUE_IDS` へ。判断に迷う場合は公開しない(疑わしきは非公開)。
2. **営業状況の裏取り**。閉店・移転の形跡(電話番号が別店舗に転送される、出典が「閉店」「掲載保留」表記、同一住所に別店の【閉店】ページがある等)がないか。裏取りできない場合は `UNVERIFIED_VENUE_IDS`(店舗ページに「営業状況を確認できていません」注記を出し、未確認の営業時間は載せない)。
3. **業態表記が実態と一致しているか**。
4. **タグの出典**(推測で付けない。表記ゆれを作らない)。

### 定期棚卸し(半年ごと)

- **半年に1度、全公開店舗の営業状況を再確認する**(閉店・移転・業態変更の有無)。現状は自動チェックがないため、この定期棚卸しで担保する。棚卸しで判明した是正は下記「掲載データの是正履歴」に、削除・非公開化は `data/venue-audit-log.md` に記録する。
- 次回目安: 2026年下期(前回全数調査 2026-07)。

### 削除・非公開化の記録を残す

- 削除(閉店)・フェーズ2移動した店は、**店名・住所・削除/移動の理由・根拠URL・日付**を **`data/venue-audit-log.md`** に記録する(誤削除の検証・問い合わせ対応・再判断のため)。
- **求人サイト由来のシグナル**(バイトル・体入ショコラ等の求人内容)は、業態・接待性の**分類の手がかりとして参照してよいが、店舗ページには表示しない**(`sources` にも載せない)。参照した事実と根拠は `data/venue-audit-log.md` の調査メモに残す。
